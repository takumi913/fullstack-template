# Postmortem 功能回顾

## 触发条件

- 用户说"功能回顾"、"merge 前检查"、"检查是否影响已有功能"
- 用户完成了一个较大的功能开发
- 准备将分支合并到 main

## 与 postmortem-check 的区别

| 项目 | postmortem-check | postmortem-review |
|------|------------------|-------------------|
| 范围 | 单次提交/PR | 整个功能分支 |
| 深度 | 模式匹配 | 功能影响分析 |
| 场景 | 日常提交 | 重要里程碑 |

## 工作流程

### 1. 收集功能全貌

```bash
# 获取分支的所有提交
git log --oneline origin/main...HEAD

# 获取所有变更文件
git diff --stat origin/main...HEAD

# 获取完整 diff
git diff origin/main...HEAD
```

### 2. 功能影响分析

识别这个功能涉及的核心模块：

- **API 层**：`api/` 目录的变更
- **前端**：`web/` 目录的变更
- **数据层**：`pkg/` 目录的变更
- **配置**：`configs/` 的变更
- **基础设施**：`docker-compose.yml`, `Makefile` 等

### 3. 历史问题关联

读取所有 `postmortem/PM-*.md`，检查：

1. **直接关联**：变更的文件是否出现在历史 postmortem 中
2. **间接关联**：变更的模块是否有历史问题记录
3. **模式重复**：是否引入了与历史问题相似的代码模式

### 4. 功能完整性检查

确认：
- [ ] 新功能是否有对应的测试
- [ ] 是否更新了相关文档
- [ ] 是否考虑了边界情况
- [ ] 是否影响现有 API 的向后兼容性

### 5. 输出报告

```markdown
## 📋 功能回顾报告

### 功能概述
[基于 commit 历史总结这个功能做了什么]

### 变更统计
- 修改文件数：X
- 新增行数：X
- 删除行数：X

### 模块影响

| 模块 | 变更文件数 | 风险等级 | 说明 |
|------|------------|----------|------|
| API | X | LOW | ... |
| Web | X | NONE | ... |

### 历史问题关联

| Postmortem ID | 关联度 | 说明 |
|---------------|--------|------|
| PM-xxx | 直接 | 修改了相同文件 |

### 检查清单
- [ ] 测试覆盖
- [ ] 文档更新
- [ ] 向后兼容
- [ ] 性能影响

### 建议
[综合建议，是否可以安全 merge]
```

## 最佳实践

- 对于大功能，建议分阶段 review
- 发现高风险关联时，主动建议创建新的 postmortem 预防措施
- 如果功能涉及历史问题区域，额外强调测试覆盖
